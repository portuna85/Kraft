# ?�� Backend Improvements

## 1. Build, Dependency, and CI Quality Gates

> **Priority:** ?�� **High** for CI enforcement &nbsp;·&nbsp; ?�� **Medium** for SBOM and digest pinning

<details open>
<summary><strong>Current state</strong></summary>

The Gradle build uses Java 25, Spring Boot 4.0.5, Checkstyle, SpotBugs, JaCoCo, and performance smoke tests. This is a strong foundation. However, `strictStatic` and `strictCoverage` are controlled by Gradle properties and are not enforced by default. The Dockerfile builds the production JAR with `-x test`, which is efficient but risky unless CI always runs tests and static checks before image creation.

</details>

> ?�️ **Risk** ??A production image can be built even when tests, coverage, Checkstyle, or SpotBugs would have failed in strict mode. This creates a gap between local convenience and deployment confidence.

**Recommended improvements**

- Add a CI workflow that **always** runs:

  ```bash
  ./gradlew clean check -PstrictStatic=true -PstrictCoverage=true
  ./gradlew performanceSmokeTest
  ```

- Keep `-x test` in the Dockerfile only if the pipeline guarantees that `check` has already passed before image build.
- Publish JaCoCo XML/HTML artifacts from CI.
- Add dependency vulnerability scanning such as **OWASP Dependency-Check**, **Snyk**, **Trivy**, or **GitHub Dependabot** alerts.
- Pin container base images by digest for production builds.
- Generate an **SBOM** for each release image.

## 2. Production Configuration and Startup Validation

> **Priority:** ?�� **High** for production profile and ops-token validation &nbsp;·&nbsp; ?�� **Medium** for build metadata

<details open>
<summary><strong>Current state</strong></summary>

The application has separate local and production profiles. The production profile disables Swagger/OpenAPI UI, hides health details, disables history initialization by default, and uses the real lottery API client. The default profile is `local`, and several important runtime values are driven by environment variables.

</details>

> ?�️ **Risk** ??If `SPRING_PROFILES_ACTIVE=prod` is not set in production, local settings may be used accidentally. Also, operation endpoints can technically run with an empty `kraft.security.ops.required-token`, relying only on IP allowlisting.

**Recommended improvements**

- Add startup validation that **fails fast** when production is active and any of these are true:
    - `SPRING_PROFILES_ACTIVE` does not include `prod` in a production deployment.
    - `kraft.security.ops.enabled=true` and `kraft.security.ops.required-token` is blank.
    - `kraft.public-base-url` does not match the canonical deployed host.
    - Database credentials are missing or clearly placeholder-like.
- Add an `/actuator/info` build section with Git commit, build timestamp, and application version.
- Document a minimal production `.env` template with required variables only, excluding secrets from version control.

## 3. Operations Endpoint Security

> **Priority:** ?�� **High**

<details open>
<summary><strong>Current state</strong></summary>

The operations filter protects `/ops/**` and `/admin/ops` with an IP allowlist and an optional token. The token can be supplied through `X-Ops-Token`, a `KRAFT_OPS_TOKEN` cookie, or the `opsToken` query parameter.

</details>

> ?�️ **Risk** ??Query-string tokens can leak through browser history, proxy logs, analytics logs, `Referer` headers, screenshots, and copied URLs. If cookie-based authentication is used for operation endpoints, state-changing `POST` requests also need CSRF protection or a header-only token policy.

**Recommended improvements**

- **Remove** `opsToken` query-parameter authentication.
- Accept the token through `X-Ops-Token` only, or use a short-lived secure cookie plus CSRF protection.
- Use **constant-time** token comparison:

  ```java
  MessageDigest.isEqual(
      expected.getBytes(StandardCharsets.UTF_8),
      actual.getBytes(StandardCharsets.UTF_8)
  )
  ```

- Add a production startup rule: operation endpoints must not be enabled with a blank token.
- Add reverse-proxy rules that block `/ops/**` and `/admin/ops` before traffic reaches the application unless the request comes from a trusted network.
- Add audit logs for **successful** operation actions, not only blocked attempts. Avoid logging full tokens or full query strings.

## 4. HTTP Security Headers

> **Priority:** ?�� **Medium**

<details>
<summary><strong>Current state</strong></summary>

The application sets `Content-Security-Policy`, `X-Frame-Options`, `Referrer-Policy`, `Permissions-Policy`, and `X-Content-Type-Options`. This is a good baseline.

</details>

> ?�️ **Risk** ??The current CSP allows external assets from `cdn.jsdelivr.net`. This is common and acceptable, but it creates external CDN dependency and expands the trusted script/style surface.

**Recommended improvements**

- Add `Strict-Transport-Security` at the reverse proxy or application layer once HTTPS-only behavior is confirmed:

  ```http
  Strict-Transport-Security: max-age=31536000; includeSubDomains
  ```

- Consider self-hosting Bootstrap and HTMX to reduce CDN dependency.
- If CDN usage remains, keep Subresource Integrity attributes and monitor version drift.
- Consider adding `upgrade-insecure-requests` to CSP after confirming no mixed-content dependencies exist.

## 5. Public Rate Limiting

> **Priority:** ?�� **Medium** for single-node operation &nbsp;·&nbsp; ?�� **High** if scaled horizontally

<details>
<summary><strong>Current state</strong></summary>

The public rate-limit filter uses an in-memory Caffeine cache and a fixed-window counter per `clientIp|path`. It currently applies to public `GET /` and `/fragments/**` requests.

</details>

> ?�️ **Risk** ??Fixed-window rate limiting allows boundary bursts. In-memory counters are not shared across multiple application instances. The current scope may miss future public endpoints unless explicitly added.

**Recommended improvements**

- Replace fixed-window logic with a **token-bucket** or **sliding-window** implementation.
- If the service becomes multi-instance, move rate limiting to Redis, an API gateway, or the reverse proxy.
- Add separate **stricter** limits for expensive recommendation and fragment endpoints.
- Add tests proving behavior at the window boundary.
- Add metrics for allowed, throttled, and cache-size states.

## 6. External Lottery API Client

> **Priority:** ?�� **High**

<details open>
<summary><strong>Current state</strong></summary>

The lottery API client has retries, request timeout handling, content validation, failure metrics, a circuit breaker, and parser tests. It treats `text/html` responses as "not drawn" and returns `Optional.empty()`, while logging a warning if the requested round should already exist.

</details>

> ?�️ **Risk** ??An HTML response for an already expected round may indicate bot blocking, upstream maintenance, WAF behavior, or an unexpected upstream format. Returning `Optional.empty()` can make an **operational failure look like a normal "not drawn yet" state**.

**Recommended improvements**

- Change HTML handling logic:

  | Round state | Behavior |
    |---|---|
  | **Future round** | Return `Optional.empty()` as "not drawn". |
  | **Current / past expected round** | Raise `LottoApiClientException` with a reason such as `HTML_RESPONSE_FOR_EXPECTED_ROUND` or `UPSTREAM_BLOCKED`. |

- Add a dedicated metric for upstream blocking / non-JSON failures.
- Preserve a sanitized, truncated preview of the upstream body in failure logs.
- Add an operations dashboard indicator showing the **last successful external API fetch time**.
- Add **contract tests** for:
    - valid JSON response,
    - future-round empty / not-drawn response,
    - HTML response for expected round,
    - HTTP 429/500 retry behavior,
    - timeout behavior,
    - malformed JSON.

## 7. Collection Scheduler and Concurrency

> **Priority:** ?�� **Medium** for single-instance &nbsp;·&nbsp; ?�� **High** for multi-instance

<details>
<summary><strong>Current state</strong></summary>

The collection command service uses an `AtomicBoolean` to avoid overlapping collection runs inside one JVM. It supports collecting one next round, collecting until latest, collecting full history, and collecting missing rounds.

</details>

> ?�️ **Risk** ??`AtomicBoolean` is **process-local**. If the application is ever deployed with more than one instance, two instances can collect at the same time. This can create duplicate external API calls, race conditions, or conflicting fetch logs.

**Recommended improvements**

- Use a database-backed lock such as **ShedLock** or a MariaDB advisory lock for scheduled and manual collection commands.
- Store lock owner, acquired time, and timeout in the database.
- Keep the JVM-local `AtomicBoolean` as a fast local guard, but **do not rely on it as the only lock**.
- Add metrics for skipped-overlap, lock-acquired, lock-timeout, and lock-release failures.
- Rename ambiguous variables such as `truncated` to `maxRunLimitReached` where appropriate.

## 8. Persistence, Flyway, and Fetch Logs

> **Priority:** ?�� **Medium**

<details>
<summary><strong>Current state</strong></summary>

The service uses JPA with `ddl-auto=validate`, Flyway migrations, and persistent fetch logs. Successful fetches do not store raw responses; failed fetches can store raw upstream content.

</details>

> ?�️ **Risk** ??Raw upstream responses can become large or contain unexpected content. Failure logs can grow quickly if the upstream API is blocked or unstable.

**Recommended improvements**

- Enforce maximum lengths for raw response, failure message, and normalized reason fields.
- Truncate and sanitize raw upstream body before persistence.
- Add database indexes on high-use query fields such as status, draw round, created time, and normalized failure reason.
- Confirm the retention scheduler runs in production and records deletion counts.
- Add operational documentation for Flyway repair, backup, restore, and failed migration handling.

## 9. Recommendation Engine

> **Priority:** ?�� **Medium**

<details>
<summary><strong>Current state</strong></summary>

The recommendation feature has explicit exclusion rules and metrics for latency, request count, rule rejection, duplicate rejection, rejection rate, and timeouts. Count normalization clamps requested recommendation counts to a safe range.

</details>

> ?�️ **Risk** ??Rule interactions can increase rejection rates and eventually trigger generation timeouts. Without operational visibility per rule, a rule configuration change can degrade the recommendation endpoint.

**Recommended improvements**

- Add an operations view for recommendation metrics:
    - rejection rate,
    - rejection count by rule,
    - generation timeout count,
    - p95 / p99 generation latency.
- Cache the rule list and any expensive historical data structures, invalidating them when new winning numbers are collected.
- Consider exposing a non-user-facing diagnostic endpoint for rule configuration and current thresholds.
- Add user-facing copy that makes the probabilistic limitation clear: **recommendations are heuristic and do not increase lottery odds in a mathematically guaranteed way.**

# ?�� Suggested Implementation Order

### Phase 1 ??Immediate Production Hardening

> ?�� *Do these first.*

2. Require non-empty operation token in production.
3. Remove `opsToken` query parameter support.
